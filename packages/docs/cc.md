  好问题！你一直用 /clear 其实是最贵的做法——每次都从零开始，所有文件都要重读。

  /compact vs /clear 决策指南                                                                         
   
  核心区别                                                                                            
                                                                                                    
  /clear  = 格式化硬盘，一切从零开始。下次还要花 20-40K tokens 重建基线。                             
  /compact = 把 200 页笔记压缩成 5 页摘要。关键信息还在，空间释放了。                                 
                                                                                                      
  什么时候该 /compact（大部分时候）                                                                   
                                                                                                      
  ┌───────────────────────────────┬─────────────────────────────────────┐                             
  │             场景              │          为什么用 compact           │                           
  ├───────────────────────────────┼─────────────────────────────────────┤                             
  │ 同一个模块的任务做了 30 分钟+ │ 上下文臃肿了，但理解还在            │                           
  ├───────────────────────────────┼─────────────────────────────────────┤
  │ 调试半天，找到原因了要开始修  │ 保留诊断结论，丢掉中间探索          │                             
  ├───────────────────────────────┼─────────────────────────────────────┤                             
  │ 写完一个功能，要写测试        │ 保留刚写的代码记忆，丢掉搜索过程    │                             
  ├───────────────────────────────┼─────────────────────────────────────┤                             
  │ /context 显示 >60% 容量       │ 预防性压缩，比等到 83% 自动压缩更好 │
  ├───────────────────────────────┼─────────────────────────────────────┤                             
  │ 读了大量文件后要开始编码      │ 保留文件结构理解，丢掉原始文件内容  │
  └───────────────────────────────┴─────────────────────────────────────┘                             
                                                        
  带焦点的 compact 更精准：                                                                           
  /compact focus on the auth module changes and test results
  /compact keep the API interface design, drop all debug output                                       
  /compact preserve the file structure understanding and current plan                                 
                                                                                                      
  什么时候该 /clear（少数时候）                                                                       
                                                                                                      
  ┌───────────────────────────────┬────────────────────────────┐                                      
  │             场景              │       为什么用 clear       │
  ├───────────────────────────────┼────────────────────────────┤                                      
  │ 从 auth 模块切到 payment 模块 │ 完全不相关，旧上下文是噪音 │
  ├───────────────────────────────┼────────────────────────────┤
  │ 上一个任务已经 commit 完毕    │ 干净收尾，新任务新开始     │                                      
  ├───────────────────────────────┼────────────────────────────┤                                      
  │ Claude 开始"幻觉"或混乱       │ 上下文被污染了，重来更干净 │                                      
  ├───────────────────────────────┼────────────────────────────┤                                      
  │ 切换到完全不同的项目          │ 旧项目的信息完全无用       │
  └───────────────────────────────┴────────────────────────────┘                                      
                                                        
  你的新工作流（替代总是 clear）                                                                      
                                                        
  开始任务                                                                                            
    ↓                                                                                                 
  读项目、探索代码（用 subagent）
    ↓                                                                                                 
  开始编码 ← /compact focus on the architecture understanding                                         
    ↓                                                                                                 
  写完功能 ← /compact focus on what I changed and why                                                 
    ↓                                                                                                 
  写测试、验证                                          
    ↓                                                                                                 
  commit 完毕 ← 现在才 /clear（或 /rename 后 clear，方便以后 --resume）
                                                                                                      
  对比你现在的做法：                                    
  以前：clear → 读项目(30K) → 编码 → clear → 读项目(30K) → 测试                                       
  花费：60K tokens 光在重复读项目                                                                     
                                                                                                      
  改后：读项目(30K) → compact → 编码 → compact → 测试 → clear                                         
  花费：30K 读一次 + compact 成本(~5K) = 35K                                                          
  节省：~40%                                                                                          
                                                                                                      
  实操小贴士                                                                                          
                                                                                                      
  1. 养成看 /context 的习惯 —— 超过 50% 就可以考虑 compact                                            
  2. compact 前先 /rename —— 给会话取个名字，万一需要回溯
  3. compact 时写清楚焦点 —— 别光打 /compact，告诉它保留什么                                          
  4. 只有真正换任务时才 /clear —— 大部分时候 compact 就够了
